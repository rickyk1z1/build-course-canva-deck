# Build Course Canva Deck

Create a complete recording presentation from a course knowledge outline or complete lecture script. The Canva deck is the only deliverable. The target is a chaptered course deck that reads like a careful human instructor built it: first read the source to identify the real teaching chapter nodes or script teaching units, then make those units the deck spine with fixed section-cover pages. Page logic is coherent, each visual teaches the current point, and a zero-basis learner understands the page without guessing. Treat the accepted `影像基础参数` deck as the content-depth and visual-quality baseline, not as reusable subject matter. Use the bundled `DAHM5fsVEB0` template profile only when the user has not supplied a different template.

The skill leans on three references plus a checklist. Read them when their stage is reached:
- [references/content-and-source.md](references/content-and-source.md) — source intake, mode contract, deck-level chapter spine, teaching standard, curriculum boundary, coverage ledger.
- [references/visual-and-design.md](references/visual-and-design.md) — structural page layouts, per-slide visuals, source-image priority, case-image stage execution, generated-image teaching specificity, template fidelity (layout language first).
- [references/roles-and-delivery.md](references/roles-and-delivery.md) — the four roles, brief contract, director acceptance lock, Canva delivery.
- [references/non-regression-checklist.md](references/non-regression-checklist.md) — the director's mandatory pre-delivery learner review.

## Non-Negotiable Checkpoint

1. Inspect every supplied source before authoring.
2. If sources conflict or overlap, ask the user to choose one authoritative source. Do not merge them by assumption.
3. After inspecting the authoritative source, ask the user to reply exactly with `细纲`, `粗纲`, or `讲稿`. Do not infer the mode from length, hierarchy, file type, or apparent detail. Do not proceed until the user chooses.
4. Record the choice in `source-map.json` with `validate_source_map.py --mode detailed|sparse|script --write`.
5. Perform a source-structure reading pass before authoring: for outlines, inspect the whole source tree, identify which nodes are real teaching chapters, and record the ordered `chapter_spine`; for complete lecture scripts, first zone sections by `source_role`, review `source_role_confidence` and extraction warnings, then identify source-faithful learner-content teaching units and record the ordered `chapter_spine`. Do not assume root children, second-level headings, natural paragraphs, or automatically inferred roles are correct when they are merely containers, course titles, framing questions, administrative wrappers,口播过渡, repeated script phrasing, visual-case briefs, or recording notes.
6. Discover the lesson's curriculum position. If the workspace does not reveal it, ask the user instead of inventing it.

## Mandatory Four-Role Workflow

For any request to build or substantially revise a course deck, use the four proposal-only roles under one director. There is no single-orchestrator authoring fallback for deck builds.

```text
总导演 / build-course-canva-deck
├── 课程统筹师
├── 原稿场记
├── 课堂编剧
└── 视觉分镜师
```

The director owns all durable writes and external actions (`source-map.json`, `curriculum-context.json`, `deck-spec.json`, generated assets, PPTX build, audit, Canva import/edit, final approval), plus process review after each proposal and final learner-facing review. Each worker is proposal-only, carries its own standard, and returns a compact handoff plus a `self_check` backed by concrete evidence. Do not create a separate reviewer role.

If subagents cannot be dispatched, runtime policy requires user authorization that has not been given, or the user disables workers, stop after safe source/mode intake and report the blocker. Read [references/roles-and-delivery.md](references/roles-and-delivery.md) before dispatching.

## Required Workflow

1. Create an external scratch workspace. Run `scripts/extract_source.py` to create `source-map.json`; use `--source-kind script` when the authoritative source is a complete lecture script. For PDFs, render and visually inspect every relevant page.
2. Complete the source and mode checkpoint above.
3. Create `curriculum-context.json` and lock the lesson's module, prerequisites, downstream lessons, shared terms, and neighboring topics that must stay out of scope.
4. Create four scoped worker briefs and dispatch in the staged order from the roles reference.
5. Build a deck spine from the approved `chapter_spine`: optional course cover, one `lesson-overview` total-introduction page, then for each selected chapter node in source order one fixed `section-cover` page followed by that chapter's content pages, and one final `summary` page. Section-cover small text previews that chapter node's immediate child headings or adjacent child-heading groups as upcoming knowledge points; it must not reveal the chapter's conclusions. In simple outlines the chapter nodes may be root children/second-level headings; in wrapped or deeply nested outlines they may start at third, fourth, or deeper levels after the source reading pass proves that is the real teaching structure.
6. Build a source coverage matrix in source order: every valid included node exactly once, with root and non-chapter wrapper/framing nodes covered by overview, approved chapter heading nodes covered by section-cover pages, and descendant nodes covered inside their chapter. In script mode, included nodes are learner-content/script-unit nodes; `metadata`, `structure_seed`, `visual_case_brief`, and `recording_note` nodes remain available for planning but are not learner-page coverage obligations.
7. Create `deck-spec.json`. Write `course.chapter_spine` as the ordered chapter node list used for section covers. Before choosing final layouts, write one learner-facing screen-copy layer that follows the approved deck spine and source order, plus a `screen.teaching_expansion` object for every normal knowledge slide. This object is the only reservoir for filling text areas: it records the learner takeaway, self-contained explanation, example/judgment cue, and whether the wording is sparse-mode vertical expansion, detailed-mode clarification, or script-mode distillation. Source order, path, coverage, curriculum exclusions, review evidence, and production strategy live in `source_node_treatments` or notes for the producer, never in rendered slide text. Optional `speaker_notes` are short internal transition hints only and must not preserve the complete script by default.
8. Create the visual plan with a per-slide asset decision pass, not a deck-level "enough images" judgment: structural pages use their fixed layout families; every normal knowledge slide first uses source case images anchored to that node when available, otherwise chooses a generated teaching case image unless the point is intrinsically a flow/table/relationship that is clearer as an editable diagram/table. Source case images follow their extracted `source_node_id` / `source_path`; do not visually infer a different knowledge point for an image that is already attached to an outline node. The director must execute `course.image_generation_tasks` for generated-image pages before PPTX build; do not leave ordinary text-heavy pages as `editable-diagram` merely because other parts of the deck have many source images. Generated-image execution must follow and record the route chain `gpt-image-2 -> imagegen -> deterministic-svg fallback`; `user-provided` is valid only for a real user-supplied or user-selected asset, never for locally drawn fallback art.
9. Build the editable PPTX with `scripts/build_deck.mjs` and `@oai/artifact-tool`.
10. Run `scripts/audit_deck.py`, render every slide, create a contact sheet, and fix all mechanical errors. A passing audit is never approval by itself.
11. Perform the director's final learner review against [references/non-regression-checklist.md](references/non-regression-checklist.md) and record it in `交付前自检.md` before Canva import.
12. Run the Canva access preflight, import the verified PPTX as a new Canva design, and leave the source template unchanged.
13. Verify every Canva page, show the complete preview, ask for one final approval, then return the link after page-count and forbidden-language checks pass.

## Hard Boundaries

- Keep the source's teaching order and hierarchy. In detailed mode the mind-map path is the course structure; do not flatten levels into a reorganized script.
- In script mode, treat the complete lecture script as the authoritative teaching source, but distill it into courseware instead of copying it paragraph by paragraph. Preserve teaching order, core judgments, examples, metaphors, and boundaries; remove greetings,口播过渡, repeated phrasing, recording reminders, and setup chatter. Do not render phrases such as `根据讲稿整理`, `这一段讲的是`, or `讲稿整理` as learner-facing copy.
- In script mode, respect source zoning but treat automatic zoning as a draft. `learner_content` drives screen copy and coverage; `structure_seed` only helps choose the chapter spine; `visual_case_brief` only guides case images, generated images, and recording-material planning; `metadata` only informs curriculum context; `recording_note` only guides production. If a script lacks clean section headings or mixes functions in one paragraph stream, the director/原稿场记 must classify by content function before authoring. Do not turn non-learner zones into required slide bullets.
- Preserve the deck spine: the first non-cover page is a total lesson overview; each approved `chapter_spine` node gets its own fixed section-cover page before its content; no chapter content may appear before its section cover; the final page is a summary.
- On section-cover pages, bullets/small text are a source-ordered overview of that chapter node's direct child headings or adjacent child-heading groups. They are not conclusions, takeaways, promises, or value judgments. Record their source anchors in `section_preview_items`; do not count these child nodes as covered until their real content pages.
- Replace the old left-footer text `线上录课课件` with `framework_progress_label` on normal content pages. The value is the current approved chapter heading for the knowledge being taught. Structural pages (`cover`, `lesson-overview`, `section-cover`, `summary`) may omit this left-footer progress text.
- Preserve detailed-outline sibling lists as complete visible lists; split slides if they do not fit.
- Do not invent a teaching label unanchored to the current node, an ancestor path, or a stated relationship. Generic labels such as `对比 A`, `结构顺序 A`, `左侧`, `方案 A/B` are failed screen copy.
- Do not render producer-facing source evidence as courseware. `本页顺序`, `对应节点`, `来源路径`, `source_node`, `screen_evidence`, coverage notes, and production language are metadata, not learner copy.
- The renderer must not invent fallback learner-facing prose when authored screen copy is missing. If `caption`, `explanation`, or bullets are absent, omit the optional visual/text element or fail earlier in authoring; never display construction/process phrases such as "how the lesson unfolds", "enter each section", or any wording that explains how the deck is organized for the producer.
- Layout empty space must be filled only with learner-facing teaching content from `screen.teaching_expansion`: definition, cause, relationship, example, misconception, judgment cue, script-distilled teaching point, or image interpretation. Never fill empty space with brief instructions, scope reminders, source coverage evidence, page-construction logic, script-processing commentary, or other producer-facing text. In sparse mode, expand vertically inside the current source node; in detailed mode, clarify and structure the source without adding new branches; in script mode, distill the current script unit into screen copy that teaches directly.
- Curriculum boundary and neighboring-course exclusions are producer constraints, not learner-facing bullets. Do not render negative scope promises such as `不进入软件按钮`, `不讲快捷键`, `不涉及调色`, or similar "we will not cover X" wording. Convert boundaries into positive learner value only when useful, otherwise keep them in `curriculum-context.json`, scope checks, and delivery notes.
- Treat every deck as one component of the same self-media and editing curriculum. Preserve shared terminology, prerequisites, difficulty progression, and lesson boundaries; do not duplicate a neighboring lesson's main task.
- Put definitions, explanations, examples, and visual interpretation on the slide. Do not create question-only, keyword-only, or separate lecture-notes deliverables.
- If a layout cannot display every required sibling item, block, or evidence phrase, split the slide or choose another layout. Never let the renderer drop, shrink, or hide content.
- Keep source images inside knowledge pages as teaching units. Every non-thumbnail source image extracted from the authoritative source is presumed to be a usable course case image and must appear on a normal knowledge page as `source-image` or `redrawn-source-image`; do not triage it away as merely weak, duplicate, stylistic, or optional. The source image's outline attachment is the authoritative teaching anchor: `extract_source.py` must preserve the image's `source_node_id` / `source_path`, and the visual plan must place it only on a slide covering that node, an ancestor, or a descendant. If the anchor is missing, fix extraction before authoring; do not guess placement by inspecting the picture. If it is dense, repeated, or low-resolution, crop, pair, sequence, or redraw it without changing the example logic. Do not raster-composite several source images into one small montage. Case-image page execution is image-first for `source-image`, `redrawn-source-image`, and `generated-image`: reserve one shared case-image stage before placing text, normally at 60%+ of the slide, and compress or split the text around it. The stage is a designed field, not the image's exact box: apply graphic-design judgment to give the actual image optically sufficient inner breathing room, aligned with the template's margins and negative-space rhythm. Render only the title and short learner-facing caption on the case-image page; do not place explanation or bullet text on top of the image stage. Use one large case image per slide by default; two may share a slide only inside one shared large case-image stage for a direct comparison/sequence, with their own `visuals[].source_image_id` when they are source images. Do not add separate backing cards under each image when one shared stage is enough. If three or more anchored/source or generated case images are needed, split pages or reduce text so each image remains readable.
- Deck length follows teaching units, source nodes, source images, and readability, not the template's page count.
- Template fidelity means the template's colors, fonts, type scale, alignment axes, spacing, density, color mass, and overall layout language reproduced as editable composition. Do not reduce template analysis to square modules or force every element into box grids; element geometry is flexible when the overall composition remains coordinated and template-like. Do not copy Canva's native vector/shape elements; there is no native-element reuse in this skill.
- A case image must visibly teach the current source point and its enumerated ideas. For image pages, record `case_visual_map` linking visible slide points to concrete image details. Pure text pages must first consider a generated case image or editable diagram and use `text-only-exception` only with a concrete bypass reason. Generated case images use the same dominant case-image stage as source case images; they must not be rendered as small decorative illustrations beside oversized text. Generated images carry no baked-in Chinese, UI labels, watermarks, or promotional text; labels stay editable slide text.
- A normal knowledge slide with no source image is not automatically an editable-diagram slide. Use `generated-image` for concrete scenarios, before/after judgment, wrong/right examples, metaphors, misconceptions, consequences, audience reactions, or workflow artifacts that a learner can recognize visually. Use `editable-diagram`/`editable-table` only when the teaching object is the relationship, sequence, axis, matrix, or label-heavy structure itself. Each `text-only-exception` and each non-generated no-source visual must record the positive reason a generated case image would teach worse.
- Generated-image pages must record the actual image route in `visual_plan` and matching `course.image_generation_tasks`: `gpt-image-2` first, `imagegen` only after GPT Image 2 fails or is unavailable, and `deterministic-svg` only after both model routes fail/unavailable or when `fallback_reason_type: "diagram-clearer"` explains why deterministic SVG/PPT teaches the knowledge point more clearly than model imagery. Do not mark locally produced SVG/PPT fallback art as `user-provided`. `user-provided` requires `user_provided_asset: true` and a `user_asset_source` naming the user's supplied/selected file.
- When any quality issue appears, run the director acceptance lock: mark the deck not approved, trace to the earliest responsible stage, fix it, rebuild, and re-run the non-regression checklist.
- Do not modify or overwrite the original Canva template.

## Resources

- `scripts/extract_source.py`: extract a canonical source map from supported formats.
- `scripts/validate_source_map.py`: validate hierarchy and record the user-declared mode (`细纲` / `粗纲` / `讲稿` as `detailed` / `sparse` / `script`).
- `scripts/audit_deck.py`: deterministic guard for coverage, source order, per-node screen evidence, mode, scope, hierarchy flattening, cross-section mixing on a single knowledge page, forbidden terms, meaningful structured labels, and PPTX/layout checks. Structural hard errors only — it does not judge teaching quality. It also emits a non-blocking geometry-repetition warning when a section runs ≥3 consecutive content pages with the same layout/image/bullet/block signature, so the director knows which pages to inspect on the contact sheet; the warning is a pointer, never approval.
- `scripts/build_deck.mjs`: generate editable 16:9 slides using the selected template profile.
- `scripts/make_contact_sheet.py`: create a labeled full-deck review sheet.
- `references/`: the three references and the non-regression checklist listed above.
- `assets/template-reference/`: immutable visual reference for Canva template `DAHM5fsVEB0` when no other template is selected.
- `assets/golden-layouts/`: approved knowledge-page composition references.
